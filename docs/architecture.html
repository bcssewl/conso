conso — Architecture

← Planning hub · the system design we build to

conso is a native macOS maintenance app — clean, optimize, analyze and monitor your Mac — built clean-room in Swift/SwiftUI for macOS 26 on Apple Silicon. It ships as one non-sandboxed app plus a small privileged helper, with a three-theme design system. This document is the contract the implementation follows.

SwiftUI · macOS 26Apple SiliconNon-sandboxed Clean-room (no GPL)Personal-first signing3 themes · light/dark

1 · Principles

  • Clean-room & original. All logic written fresh; we call only standard OS tools (brew, mas, powermetrics, IOKit, du). No Mole code, no GPL obligations, none of their globe/solar-system motif.
  • Safe by default. Everything is previewed before deletion; protected paths are never touched; every destructive op is logged and reversible where possible (Trash, not rm, when feasible).
  • Trust nothing cached — verify live. Permissions and helper state are checked on every launch and pre-flighted before any operation. Destructive ops fail loud (a clean must never silently skip); read-only/optional pillars degrade gracefully to a visible "unavailable" state. (See §4.)
  • Small, testable units. Each pillar is an independent module behind a protocol, so the UI, the engines, and the helper can be reasoned about and tested in isolation.

3 · Theming layer

A ThemeStore drives the whole UI from semantic tokens (background, card, accent, text tiers, materials, corner radius, density, motion) — never hard-coded colors. Three built-in themes, each honoring system light/dark:

  • Modern Native — macOS 26 Liquid Glass: neutral materials, edge lensing, restrained accent.
  • Pro Dark — Vercel/Geist: true black, hairline borders, near-monochrome, mono numerics.
  • Character — Liquid-glass base + the conso mascot (the running figure in the HUD).

The HTML mockups in /mockups are the visual spec; the SwiftUI implementation maps these tokens to a Theme struct + environment value. The logo is a per-theme asset (blue / white / mascot), the app-icon tile is the Dock icon.

3.1 · Accessibility — a cross-cutting requirement, not a Phase-5 afterthought

A glass-heavy UI fails accessibility audits unless designed for it from the start. We honor the SwiftUI accessibility environment values throughout:

  • Reduce Transparency (accessibilityReduceTransparency) — the ThemeStore exposes an opaque material branch: when set, glass/blur is replaced with solid token-driven fills.
  • Reduce Motion (accessibilityReduceMotion) — drop or shorten animated transitions, sparkline tweens, and the mascot run cycle.
  • Increase Contrast (colorSchemeContrast) — stronger borders/text tiers via the contrast-aware token set.
  • Dynamic Type — text scales with the user's preferred size; no fixed pixel fonts in semantic UI.
  • VoiceOver — every control, metric, and destructive action carries an accessibility label + value/hint.

4 · macOS realities — hard constraints, not afterthoughts

Source: distilled from docs/macos-lessons.md (hard-won from a prior shipped macOS app). For a maintenance app whose core value depends on Full Disk Access + a root helper, these are make-or-break.

4.1 · Code signing — the #1 time sink

App signing and helper signing are two different problems — don't assume one cert covers both.

App (for TCC / Full Disk Access persistence):

  • TCC binds a permission to the binary's code identity, not its name. A self-signed/ad-hoc app effectively binds the grant to the exact binary; change the binary and the grant is silently voided — System Settings still shows it "on."
  • Bake ONE stable self-signed certificate into the project and sign every dev build with it. Never ad-hoc (ad-hoc mints a fresh identity per build → Full Disk Access resets every rebuild → the "re-grant on every launch" trap). For the app's TCC persistence, a stable self-signed cert is fine.
  • Install to /Applications and run only from there. Two copies of the same bundle id → the grant attaches to whichever prompted; the other silently records nothing. No dist/-vs-installed drift.
  • Bundle id is fixed (com.conso.app) across dev and release — UserDefaults, Keychain and TCC are all per-bundle-id and diverge silently otherwise.
  • Untangle a bad grant with tccutil reset SystemPolicyAllFiles com.conso.app, then relaunch and re-grant.

Helper (for the SMAppService privileged daemon): a hand-rolled self-signed cert is not enough. SMAppService daemon registration is not supported with ad-hoc and is unproven/undocumented with a self-signed cert that has no Team ID. The realistic minimum is an Apple-issued certificate with a Team ID — a free Apple Development certificate (any free Apple ID, still $0, not the $99 Developer Program) — used to sign the app and the helper identically. This must be proven before anything depends on the daemon (see roadmap Phase 0 spike).

Fallback if the daemon proves unworkable: the genuinely root-only tasks are few. An Authorization Services one-shot privileged tool, or osascript … with administrator privileges per operation, can avoid a persistent daemon entirely (at the cost of a prompt per elevated action).

4.2 · Permissions model & UX

PermissionWhyHow we verify (live)
Full Disk AccessScan/clean/treemap protected foldersAttempt to read a protected path conso actually needs (e.g. a protected user-data folder it scans, such as ~/Library/Mail or ~/Library/Safari) — not the SIP-protected TCC database, and not the toggle. Distinguish exists-but-denied (EPERM/EACCES) from absent (ENOENT) — only the former means FDA is missing.

NotificationsTask-finished alerts (optional)UNUserNotificationCenter settings.
  • Ask everything up front in onboarding ("Enable All") — never lazy per-feature prompts that hide a missing grant until mid-task.
  • Verify on launch & pre-flight before every operation. Doctor surfaces anything missing with a button to the exact Settings pane.
  • Fail loud — but scoped to destructive ops. A destructive op (clean, uninstall) must never silently skip protected files because FDA lapsed — detect and stop; that's the worst bug class. Read-only / optional pillars (Updaters, deep sensors, a helper hiccup — including the known macOS 26 SMAppService XPC bug) should degrade gracefully: show a visible "unavailable" state, not hard-block the whole app.

4.3 · Entitlements, sandbox & usage strings

  • Non-sandboxed — broad disk/system access can't run sandboxed → no Mac App Store; Developer-ID/direct only.
  • Hardened runtime with com.apple.security.cs.disable-library-validation only if we load third-party dylibs; com.apple.security.automation.apple-events if we script Finder (e.g. empty Trash).
  • Info.plist usage strings or instant SIGABRT: NSAppleEventsUsageDescription (Finder/automation), NSUserNotificationsUsageDescription (notifications). Add each before requesting that capability.

4.4 · Distribution & packaging

  • Personal-first: $0. The app uses a stable self-signed cert for TCC persistence; the helper requires a free Apple Development cert (Team ID) for SMAppService (see §4.1) — both still $0, no Developer Program needed for personal use.
  • First-launch friction is worse on macOS 26 than just "Open Anyway." For an un-notarized app it's now a multi-dialog dance (an initial block, then System Settings → Privacy & Security → Open Anyway, then an admin-password confirmation; right-click→Open was removed in macOS 15). Tolerable for yourself once — but the moment anyone else installs it, this is unacceptable, so notarization (next bullet / roadmap Phase 6) is effectively required for sharing.
  • Later (if shared): Apple Developer Program ($99/yr) → Developer ID + notarization; add Sparkle and bump CFBundleVersion every release (Sparkle compares build number, not marketing version).
  • SwiftPM resource bundles at the .app root so Bundle.module resolves; launch dev builds via open (not exec'ing the inner binary, which breaks usage-string attribution); @main entry is a synchronous main().

5 · Privileged helper

A minimal, audited XPC helper registered with SMAppService (modern replacement for SMJobBless), installed once with a single admin prompt. It exposes a narrow, typed interface and does only what genuinely needs root:

  • Purge caches/logs under root-owned/system paths the app user can't touch.
  • Run maintenance routines that require elevation (some periodic/cache rebuilds, permissions repair) and elevation maintenance.

Thermal & fan reads do NOT need the helper and have moved to the unprivileged app: the Darwin notification com.apple.system.thermalpressurelevel, ProcessInfo.thermalState, IOReport sampling, and SMC fan-RPM reads all work without root. Reserving the helper for genuinely root-only work keeps its attack surface minimal. (powermetrics still needs root, but for conso's metrics it isn't required — IOReport covers it from the app.)

Everything else stays in the unprivileged app.

5.1 · XPC security

  • Validate the caller from the audit token, never the PID (PID is reusable/spoofable). Set NSXPCListener / NSXPCConnection.setCodeSigningRequirement(_:) with a requirement pinned to our Team ID + bundle id, evaluated against the connection's audit token.
  • Takes structured commands (a typed interface), never arbitrary shell, and logs every action to ~/Library/Logs/conso/operations.log.
  • The helper must never set com.apple.security.cs.disable-library-validation — root + unvalidated libraries is a local-privilege-escalation gift. Keep it a single, hardened, library-validated binary with no third-party dylibs.

Fan control is intentionally absent — not possible on Apple Silicon; we read fan RPM/thermals only (and now from the app, not the helper).

6 · Data flow — a clean operation

Scanner.scan()  ──►  Plan (categories, sizes, per-item paths)  ──►  UI preview (user selects)
                                                                          │
                                            user confirms ◄───────────────┘
                                                  │
                  Cleaner.execute(plan):  protected-path guard ► Trash-first ► (root items ► helper via XPC)
                                                  │
                              per-item results ► OperationLog ► UI summary ("11.6 GB reclaimed")

Read-only pillars (Status, Analyze) follow the same shape minus the mutation step. Status/HUD sampling runs on a timer with a 60-second ring buffer per metric.
