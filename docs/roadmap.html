conso — Roadmap

← Planning hub · the phased build plan

A phased plan where every phase ends in a runnable, signed app installed to /Applications — never a half-wired branch. The hard-won macOS lessons (stable signing, live permission checks, run-from-/Applications) are applied from Phase 0, not bolted on later. Phases are described by deliverable, not dates; each one earns its keep before we move on.

Every phase shipsLessons-firstSelf-signed → notarize-ready 7 phases (6 optional)Runnable always
Operating rule: the main branch is always installable and grant-stable. We add a pillar only when the previous one runs end-to-end from /Applications with permissions verified live. No "it'll work once we wire it up" — if it isn't running, the phase isn't done.

2 · Phase-by-phase

Phase 0 — Foundation — get the traps out of the way first

Goal: a correctly-signed, themeable app shell that survives rebuilds without losing future permission grants.

Ships: an empty-but-themeable conso.app in /Applications — opens, switches all 3 themes + light/dark, shows the menu-bar item. Nothing functional yet, everything correct.

  • Xcode project (not a bare SwiftPM executable) with a fixed bundle id com.conso.app for dev and release alike.
  • Bake ONE stable self-signed certificate into the build config and sign every build with it — never ad-hoc. (Fine for the app's TCC persistence; the helper needs more — see the spike below.)
  • A make install / build-phase dev workflow that copies to /Applications and launches via open — the canonical run path from day one.
  • App shell: MenuBarExtra (live HUD host) + main window with the top-centered glass pill nav from the mockups.
  • ThemeStore with semantic tokens, all 3 themes (Modern Native / Pro Dark / Character) honoring system light/dark, plus the per-theme logo + app-icon assets. ThemeStore exposes an opaque material branch for Reduce Transparency from the start.
  • Skeleton view-model + engine protocols (empty implementations) so later phases slot in behind interfaces.
  • SMAppService daemon registration spike — a throwaway helper that we actually register and XPC-ping, to prove the privileged-daemon mechanism before any pillar depends on it. Sign app + helper identically with a free Apple Development cert (Team ID) — a hand-rolled self-signed cert (no Team ID) is unproven for SMAppService daemons (and ad-hoc is unsupported). If the free Development cert turns out insufficient on macOS 26, Developer ID moves earlier (out of the optional Phase 6). Fallback if the daemon proves unworkable: the few root tasks can go through an Authorization Services one-shot tool or osascript … with administrator privileges instead of a persistent daemon.

Lessons applied: stable cert a named self-signed cert keeps TCC grants across rebuilds (no "re-grant every launch"). Team-ID cert for helper the SMAppService daemon needs an Apple-issued (free Development) cert, proven by spike. fixed bundle id UserDefaults/Keychain/TCC are per-bundle-id — pin it now. run from /Applications launch via open from /Applications only, so grants attach to the one true copy. Doing this in Phase 0 means nothing we build later has to be re-signed or re-pathed.

Phase 1 — Status + HUD — the fastest path to a genuinely useful build

Goal: a live system monitor that's worth keeping open, before we touch a single permission prompt.

Ships: a real-time Status dashboard + menu-bar HUD — installable and useful with zero special permissions.

  • Metrics engine: CPU/GPU/mem/disk/net/battery/thermal via IOKit, IOReport, host_statistics, sysctl (all unprivileged) on a sampling timer with a 60-second ring buffer per metric. Thermal/fan reads are unprivileged in the app — ProcessInfo.thermalState, the com.apple.system.thermalpressurelevel Darwin notification, IOReport, and SMC fan RPM — no helper needed. System-wide GPU % has no public API on Apple Silicon: ship it best-effort (private IOReport) or leave it out of v1.
  • Status view: health score hero, metric cards (temps, histogram, area charts), and a live process table — matching mockups/main-window.html.
  • Menu-bar HUD with sparklines, driven by the same Metrics stream.
  • First wiring of a real engine behind its protocol → proves the architecture end-to-end.

Lessons applied: deliberately permission-free, so this phase ships fast and we don't gate "useful" behind the FDA/helper work. Thermals, fans and even GPU come from the app via IOReport/SMC — the helper isn't needed for metrics at all. If a deep sensor or the helper ever hiccups (incl. the known macOS 26 SMAppService XPC bug), the affected card degrades gracefully to "unavailable", never hard-blocking the dashboard.

Phase 2 — Permissions, Doctor & Onboarding — verify-live infrastructure

Goal: the permission + helper machinery that every later destructive pillar depends on — built once, correctly.

Ships: first-run onboarding, a Doctor self-check, and a helper that installs and answers an XPC ping.

  • PermissionService with live verification: confirm Full Disk Access by reading a protected path conso actually needs (e.g. ~/Library/Mail or ~/Library/Safari), not the SIP-protected TCC database, and never by trusting the System Settings toggle or a cached "granted". Distinguish exists-but-denied (EPERM) from absent (ENOENT) — only the former means FDA is missing.
  • Onboarding flow with "Enable All" — ask for every grant up front, with a deep-link to the exact Settings pane (and the first-launch friction note: on macOS 26 an un-notarized app is a multi-dialog dance, not just "Open Anyway").
  • SMAppService privileged-helper skeleton (com.conso.helper, built on the Phase 0 spike) + XPC ping round-trip; the helper validates the caller via NSXPCConnection.setCodeSigningRequirement(_:) evaluated from the connection's audit token (never PID), pinned to our Team ID + bundle id, and never sets disable-library-validation.
  • Doctor: live self-check of permissions, helper status, signing, disk space, update health — fail-loud, each issue with a button to its fix.
  • Add Info.plist usage strings (NSAppleEventsUsageDescription, NSUserNotificationsUsageDescription) before requesting those capabilities.

Lessons applied: verify live a read of a path conso needs (not the TCC db) is the only truth, distinguishing denied from absent; the toggle and the cache both lie. ask up front no lazy per-feature prompts that hide a missing grant mid-task. pre-flight Doctor + per-op checks catch a lapsed grant before an operation, not during. audit-token XPC code-signing requirement from the audit token, never PID. first-launch friction documented in onboarding (worse than "Open Anyway" on macOS 26; right-click→Open was removed in macOS 15).

Phase 3 — Clean + Analyze — the first destructive pillar, on safe rails

Goal: reclaim space and visualize disk usage — safely, on top of the verified-permission base from Phase 2.

Ships: a working Clean flow (preview → confirm → reclaim) and an interactive disk treemap.

  • Scanner: walks caches/logs/build-junk/leftovers, sizes categories, produces a previewable plan.
  • Cleaner: executes a reviewed plan — Trash-first, protected-path guard, per-item result; root-owned paths go through the helper via XPC.
  • Clean view: reclaimable-space ring + categorized list (mockups/clean.html), reassuring copy, nothing deleted without explicit confirm.
  • DiskMap: concurrent directory walk → treemap with drill-down + largest-folders panel (mockups/analyze.html).
  • Pre-flight FDA + helper before any scan/clean; surface a missing grant via Doctor, never degrade silently.

Lessons applied: fail loud (destructive) a clean/uninstall that silently skips protected files because FDA lapsed is the worst bug class — detect and stop; this strict rule is reserved for destructive ops. degrade (read-only) Analyze/treemap and optional sensors show "unavailable", not a hard block. pre-flight verify FDA + helper live before walking, not after. Trash-first keeps every op reversible.

Phase 4 — Software + Optimize — external tools + root maintenance

Goal: keep installed software current and run the maintenance routines, splitting user-level work from helper-only work.

Ships: a Software pane (updates + uninstall) and an Optimize pane (maintenance routines).

  • Updaters: surface updates across Mac App Store (mas/softwareupdate), Homebrew (brew), Sparkle feeds, and Electron/Squirrel — shelled out via Process (mockups/software.html).
  • Uninstaller: find an app's remnants across known locations; remove app + leftovers + login items.
  • Optimizer: maintenance routines (Spotlight, DNS, Quick Look, font/Launch Services, etc.) — user-level where possible, helper for root-only tasks (mockups/optimize.html).
  • Resolve, per routine, the open decision: does it truly need root (helper) or can it run as the user?

Lessons applied: the helper takes structured commands, never arbitrary shell, logs every action to ~/Library/Logs/conso/operations.log, and is reserved for genuinely root-only work (root-owned cache purge, elevation maintenance) — thermals/fans are read by the app, not the helper. Fan control stays absent — Apple Silicon exposes read-only thermals.

Phase 5 — Extras + polish — round out and harden

Goal: the smaller pillars plus the cross-cutting polish that makes it feel native.

Ships: Privacy monitor, screen/keyboard utilities, the Quick Actions tray, reclaimers, a duplicate finder, a full Settings pane, accessibility, and a performance pass.

  • Privacy: camera/mic in-use detection in the HUD (CoreMediaIO / system log).
  • ScreenTools: Keep Awake (caffeinate-style, duration + menu-bar toggle), Clean Keys (lock keyboard/trackpad ~10 s with a countdown overlay so you can wipe the keys), Clean Screen (full display-wipe overlay).
  • Quick Actions tray in the menu-bar HUD: Keep Awake · Clean Keys · Clean Screen · Show Hidden Files · Empty Trash.
  • Large hidden-items reclaimers (Analyze/Clean extension): APFS local snapshots, iOS device backups, Mail attachments — recovery data, default-unchecked and clearly labelled.
  • Duplicate & large/old-file finder (extension of Analyze): size + content-hash matching.
  • Full Settings: theme, density, launch-at-login, notifications, helper management.
  • Accessibility wired through SwiftUI environment values: Reduce Transparency (opaque material branch in ThemeStore), Reduce Motion, Increase Contrast, Dynamic Type, and VoiceOver labels on every control + metric — an ongoing requirement since Phase 1, hardened here. Plus a performance pass on sampling + the directory walk.

Lessons applied: any new capture needs its usage string + matching hardened-runtime entitlement before it's requested, or the app SIGABRTs on first use. a11y is cross-cutting the opaque/reduced-motion/contrast branches were designed into the ThemeStore in Phase 0, not retrofitted now.

Phase 6 — Share-ready — optional, only if we distribute

Goal: turn the personal build into something others can install and auto-update. Personal-only is fine un-notarized — but on macOS 26 first launch for others is a multi-dialog block + admin password, so notarization is effectively required the moment anyone else installs it.

Ships: a notarized, Developer-ID-signed, auto-updating build.

  • Apple Developer Program ($99/yr) → Developer ID signing + notarization (no Mac App Store — we're non-sandboxed).
  • Sparkle auto-update wired to a release feed.
  • Bump CFBundleVersion every release, monotonically — Sparkle compares build number, not marketing version, or it says "up to date" forever.

Lessons applied: stable cert signing every release with one shared identity is what lets users keep their grants + Gatekeeper approval across updates. Until we have the account, a full notarize pipeline is wasted effort — that's why it's last and optional.

3 · Risks & how we de-risk

RiskHow we de-risk
TCC / signing resets — grants silently vanish on rebuild.One stable self-signed cert baked into config from Phase 0; fixed bundle id; tccutil reset SystemPolicyAllFiles com.conso.app as the known untangle command.
Helper signing / SMAppService unproven — daemon won't register, or XPC is insecure.Phase 0 SMAppService registration spike proves it (register + ping) on a free Apple Development cert with Team ID — self-signed/no-Team-ID is unproven, ad-hoc unsupported; if insufficient on macOS 26, Developer ID moves earlier. App + helper signed identically; caller validated via a code-signing requirement from the audit token; helper never disables library validation. Fallback: Authorization Services / osascript … with administrator privileges for the few root tasks.
FDA UX — a lapsed grant degrades a destructive clean silently.Live verification by reading a path conso needs (not the TCC db), distinguishing denied (EPERM) from absent (ENOENT); "Enable All" up front, pre-flight before every op, Doctor failing loud with a deep-link — never trust the toggle. Read-only pillars degrade to "unavailable" rather than hard-block.
Two-copies drift — grant attaches to the wrong bundle.Install-to-/Applications dev workflow from Phase 0; launch via open only; confirm the running path, never run a dist/ copy.
Scope creep — pillars half-wired, nothing shippable.The operating rule: each phase must run end-to-end from /Applications before the next starts. Permission-free Status (Phase 1) lands value early; helper-dependent pillars come only after the Phase 2 base is proven.
Open decisions carried from Architecture §7: (1) confirm bundle id com.conso.app; (2) personal-only now vs notarize-ready later (Phase 6); (3) which Optimize routines truly need the helper vs run as user (Phase 4). These are resolved at the start of implementation, not deferred indefinitely.
